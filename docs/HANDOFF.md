## Current state ‚Äî v0.15.0 "Death/Enemy Animation Pass"
- **Player death burn-up:** every character has `death-0..3` frames (FRAME_NAMES is now 19 per character; `PlayerState` gained `'death'`; STATE_DEFS entry = frameRate 8, repeat 0). The frames are NOT hand-drawn or PixelLab-final art ‚Äî **`tools/gen_death_frames.py` is the source of truth**: a deterministic Pillow post-processor (rng seeded per character id) that turns a flinch pose into tint ‚Üí ember ‚Üí char/crack ‚Üí dissolve. Flinch source = `tools/death-raw/<id>/flinch.png` when present (PixelLab hurt poses for ember/cole/kiko, gitignored raws), else the character's `jump-3` rotated ‚àí12¬∞ (classic ships the transform fallback). Re-running the script is always safe and reproducible; to restyle a death, regenerate ‚Äî never hand-edit the PNGs.
- **Wiring:** `Player.playDeath()` stops the current anim, plays `<char>-death`, and sets `deathPlayed` ‚Äî `pickAnimation` early-returns on it so nothing overrides the burn-up (no reset needed: playDeath only fires on FINAL death, after the revive branch, and each run constructs a fresh Player). `GameScene.triggerDeath` calls it right after `this.dead = true`.
- **Clip finish moved INTO the 450ms death delay:** `ClipRecorder.finish()` now runs as the first lines of the `delayedCall(450, ...)` callback (not at death time), so the recorder keeps rolling through the burn-up and shared clips END on it. `clipDone` still rides `gameOverData` (same scope). This supersedes the v0.14.0 note below claiming finish happens at death time. `completeLevel()` unchanged ‚Äî no death anim on level clears.
- **Enemies animate via timed texture cycling, not Phaser anims:** enemies stay `add.image`. BootScene loads `enemy-crawler-0..3` + `enemy-drifter-0..1` (PixelLab regens normalized by `tools/normalize_enemy_frames.py` ‚Äî union-bbox across each enemy's frames for one shared scale, crawler bottom-anchored, drifter centered, 48px canvas). EnemyManager counts available textures at construction (`while textures.exists(...)`) and cycles per-view (`frameIdx`/`frameMs`, crawler 125ms / drifter 167ms), re-asserting `setDisplaySize` after every `setTexture` (texture swaps reset frame size). **Zero detected frames = byte-identical static fallback** ‚Äî the old generated-texture path still works if the PNGs vanish.
- **Titan roar is derived, not regenerated:** `tools/gen_titan_roar.py` reads `titan.png`, brightens the warm mask (mouth/eye/crack glow), scales 6%, re-centers ‚Üí `titan-roar.png`. `BossController.start()` shows the roar for 700ms then reverts (guarded by `textures.exists` + `titan.active` so headless/fallback modes never break).
- **e2e (`e2e/anims.spec.ts`):** death frame via texture-key regex `/-death-\d$/` after a direct `triggerDeath` poke; crawler cycling via a direct `enemies.spawn()` poke (the descriptor must satisfy `PlatformDescriptor` ‚Äî `type: 'static', hasCoin: false`, NOT the older `behavior`/`coin` names); titan roar via `boss.start(0, g.runSeed, g.lava.surfaceY)` then polling `boss.titan.texture.key` roar‚Üírevert. Clips specs pass unchanged ‚Äî the +450ms finish sits inside their existing poll timeouts.
- Suite: 234 unit + 29 e2e.

## Previous state ‚Äî v0.14.0 "Highlight Clips + Social Sharing"
- **Every run auto-records a rolling highlight** (setting `settings.recordClips`, default true, backfilled by the settings deep-merge + migration-tested). On death OR level clear, Game Over shows **‚ñ∂ SHARE CLIP (C)** ‚Üí DOM overlay with looping replay, SHARE (native share sheet ‚Äî only rendered when `navigator.canShare({files})`, i.e. phones), DOWNLOAD (`lava-leap-<score>.mp4|webm`), COPY CAPTION.
- **Why two recorders:** a MediaRecorder's output is only a valid file from its own start (chunk #1 holds the container header), so "last 30 s of one long recording" yields broken files. `src/core/clipRotation.ts` (pure, unit-tested) leapfrogs two recorders on a 15 s cadence (`SEGMENT_MS`); the OLDER-running one always holds 15‚Äì30 s ending now and becomes the clip (`pickWinner`). `src/entities/ClipRecorder.ts` applies the actions to real MediaRecorders over `canvas.captureStream(30)` + a `MediaStreamAudioDestinationNode` tapped off Phaser's `masterVolumeNode` (clip audio = player's mix; HTML5-audio fallback records video-only). Mime pick prefers MP4 (`video/mp4;codecs=avc1`) and falls back to WebM. 2.5 Mbps.
- **`clipDone` rides `gameOverData`** exactly like v8's `submitDone` ‚Äî `finish()` is called at death time (NOT inside the 450 ms delayedCall), so encoding overlaps the death animation; GameOverScene `.then()`s the promise and only adds the row on a non-null clip. Survives the Cutscene hop (data object passes through). Recorder errors/timeouts (3 s bail) resolve null and tear down silently ‚Äî the game never breaks over a clip. Pause pauses recording (clips don't contain the pause screen).
- **SettingsScene refactored to key-based rows** (`ROW_DEFS` + `RowKey`) because the REC CLIPS row is conditionally present (`ClipRecorder.supported()`), which would have broken the old hard-coded index mapping. Row geometry: `250 + i*40`, tap buttons y 545, hint y 575. Any future conditional row: extend `ROW_DEFS`, never index math.
- **e2e:** `e2e/clips.spec.ts` ‚Äî MediaRecorder + captureStream work fine in headless Chromium under the existing SwiftShader flags; the clip test fetches the overlay video's blob URL and asserts >10 KB of real footage. `triggerDeath` poked directly (runtime JS ignores TS `private`).
- Suite: 232 unit + 26 e2e.

## What it is
**Lava Leap** ‚Äî an endless vertical climber (Phaser 3 + TypeScript + Vite + Vitest). Climb procedurally-generated platforms, outrun rising lava; score = height + coins. Own git repo (`master`), public at **github.com/denrod25-del/lava-leap**. Also packaged as an Android app via Capacitor and deployed to the web on Vercel.

## Previous state ‚Äî v0.12.0 "Power-up Pass"
- **v0.12.0 = a power-up economy pass**, three coordinated changes: much higher spawn rates with the rocket (orange boost) weighted as the most common kind, a full visual redesign of the pickups, and ~20% slower lava.
- **Spawn-rate increase:** `HAZARD.powerupChance` 0.03 ‚Üí 0.20. Initially shipped as 0.06 ("2x"), but the measured reality was ~0.1‚Äì0.3 power-ups in a typical run's first 2000px ‚Äî doubling a near-zero rate was imperceptible; 0.20 lands ~1.8 power-ups per first-2000px (measured). **Lesson: for frequency tuning, measure the per-run FELT rate (events per typical run), not the per-platform probability ‚Äî a multiplier on a tiny base is invisible.**
- **Weighted kind mix:** `rollPowerup` (`src/core/hazardRules.ts`) now picks from `POWERUP_WEIGHTS` ‚Äî rocket 0.55, shield/magnet/slowlava 0.15 each (was 0.4/0.2 in round one; raised on a further "more rockets" play-feedback round) ‚Äî instead of a uniform 4-way split. Still exactly 2 RNG draws per roll (the weighting only remaps the second draw), so the per-seed platform stream is unchanged; only kind assignments shift.
- **The unlock-gate arc (built, then removed ‚Äî kept here as history):** the original v0.12.0 design gated the rocket behind height 4000 in endless/daily (`rocketUnlockHeight` threaded through `LevelStream`‚Üí`LevelGenerator`, later softened to convert gated rockets into shields). Play-testing rejected it twice: the gate made the boost invisible in normal play, and the shield-conversion made blue dominate ‚Äî the opposite of the "more orange" the user wanted. All gate machinery was removed before ship (no trace remains in code; `git log` commits `aa00bc2`/`1996d5f`/`6c98e2d` hold the history). Two real bugs from that arc worth remembering anyway: the naive `height < Infinity` check is always true (an Infinity-default gate must use `Number.isFinite`), and discard-style gating silently cuts total density.
- **Lava slowdown (~36% cumulative):** `src/entities/Lava.ts` `BASE_SPEED` 55 ‚Üí 44 ‚Üí 35 and `SPEED_PER_HEIGHT` 0.012 ‚Üí 0.010 ‚Üí 0.008 across two play-feedback rounds (zone multipliers untouched). note high scores will trend higher vs. the pre-0.12 leaderboard era.
- **Pickup visual redesign (`PowerupController`):** each pickup is now a Container holding a circle plus a hand-drawn Graphics icon per kind, with a pulse tween ‚Äî **zero new PixelLab art** (trial still has 1 generation banked). Two real bugs caught and fixed during review: (1) the infinite-repeat pulse tween was never killed on container destruction ‚Äî leaked a live tween + retained object per pickup; (2) tweening the CONTAINER's scale made the Arcade body oscillate ~8‚Üí9px (`Body.updateBounds` reads container scale each preUpdate) ‚Äî fixed by tweening the **children** instead, so the collision footprint is constant by construction. Teardown consolidated into a single `removePickup(id)` helper (despawn, collect, and shutdown all route through it).
- **Tests:** 214 unit + 23 e2e green; typecheck + build clean; `0.12.0` confirmed in the production bundle.

## Previous state ‚Äî v0.11.0 "Levels Mode"
- **v0.11.0 = a real 4-level campaign**, layered on top of endless with almost no new machinery. `src/core/levels.ts` defines `LEVELS: LevelDef[]` (`id`, `title`, `zoneIndex`, `startHeight`, `bossTriggerHeight`, `bossTemplateId`) ‚Äî 4 entries, each spanning exactly 1000px (`level-1` 0‚Üí1000, `level-2` 1000‚Üí2000, `level-3` 2000‚Üí3000, `level-4` 3000‚Üí4000) and a pure `isLevelUnlocked(levelId, clearedIds)` (sequential: level N unlocked iff level N-1 is in `clearedIds`). SaveData gained `levels: { cleared: string[] }` (deep-merge + array-guarded like every other nested field).
- **The spawn-offset trick is why this was cheap:** every height-driven system in the game (`zoneForHeight`, the generator's difficulty ramp, `StoryProgress.onHeight`, `RelicPlanner`) already computes purely from `TUNING.groundY - player.y`. `LevelGenerator`/`LevelStream` gained one optional constructor param (`startHeightOffset`, default 0 ‚Äî endless/daily are byte-identical) that shifts the FIRST platform's `y` and therefore the player's spawn height. Every other system just sees a bigger `heightClimbed` on frame one and behaves correctly with zero level-awareness of its own.
- **`GameScene` reused almost entirely:** `init(data)` now accepts an optional `levelId`, resolved to a `LevelDef` (`this.levelDef`). When set: background/zone/spawn/`LevelStream` all key off `startHeightOffset` instead of hardcoded zone-0/height-0; the boss trigger in `update()` branches on `this.levelDef` ‚Äî level runs check their OWN `bossTriggerHeight`/`bossTemplateId` (never calling `bossBoundaryCrossed`), so `BOSS_BOUNDARIES`/`boss.ts` are **completely untouched** by this feature (`tests/boss.test.ts` still locks the endless array to `[1000,2000,3000]`, proving it). A 4th boss template (`titan-4`, hand-authored + `validateChunk`-checked) exists ONLY as a level-mode finale ‚Äî endless mode still has just 3 boss encounters.
- **`completeLevel()`** (new `GameScene` method, fired from the `bossPhase 'end'` handler when `this.levelDef` is set): bank coins via the existing `endRunBookkeeping`, push the cleared id into `save.levels.cleared` (idempotent), route to `GameOverScene` with `{result:'cleared', levelId, nextLevelId}` ‚Äî through `Cutscene` first if one's pending, exactly like a normal death does. **Leaderboard-gating fix:** `triggerDeath`'s auto-submit block now reads `if (leaderboard.enabled && !this.levelDef)` ‚Äî a death INSIDE a level no longer contaminates the endless leaderboard (this was a real bug the spec missed; caught while writing the plan, not by the design doc).
- **`GameOverScene`** branches on `data.result === 'cleared'`: cyan "LEVEL CLEAR" heading, "Press SPACE for the next level" (only if `nextLevelId` exists ‚Äî Level 4 shows just "ESC ‚Äî Level Select"), no leaderboard rank block ever (levels never set `playerId`). Dying inside a level restarts that SAME level (`levelId` now threads through the retry call) rather than falling back to endless, matching "restart from the top."
- **`LevelSelectScene`** (`'LevelSelect'` key, Menu hotkey `K`): the established Journal/Shop vertical-list idiom ‚Äî 4 rows, `[CLEARED ‚òÖ]` / `[PLAY]` / `[LOCKED üîí]` computed live from `save.levels.cleared` via `isLevelUnlocked`. Clicking/entering a locked row is a no-op (no scene change, no error).
- **Version-bump ordering note (confirmed again this release):** `e2e/levels.spec.ts` seeds `lastSeenVersion: '0.11.0'` ‚Äî that seed only resolves to `Menu` (not an auto-redirect to `Changelog`) once `package.json`'s version actually matches. Bumped the version + changelog entry BEFORE running the new e2e spec, same pattern the v0.10.0 handoff already documents below.
- **Tests:** 213 unit + 23 e2e green; typecheck + build clean; `0.11.0` confirmed in the production bundle.

That decision was reversed this release ‚Äî levels mode is now shipped as an optional companion mode alongside endless/daily, not a replacement. See that section for the corrected line.

## Key technical facts / gotchas
- **Architecture:** pure logic in `src/core/` (never imports Phaser; unit-tested); Phaser in `src/scenes/` + `src/entities/`. Tuning in `src/tuning.ts`. In `GameScene` the event hub is `this.gameEvents` (Phaser owns `this.events`); input source is `this.inputSrc`.
- **Build info:** `vite.config.ts` `define`s `__APP_VERSION__` (from `package.json` via `readFileSync` ‚Äî tsconfig has no `resolveJsonModule`, `moduleResolution: bundler`), `__BUILD_ID__` (git short hash via `execSync`, try/catch ‚Üí `'dev'`), `__BUILD_DATE__`; `src/build-globals.d.ts` declares them; `src/core/buildInfo.ts` re-exports with vitest fallbacks. **Bump `package.json` version each release** ‚Äî every on-screen label reads it automatically.
- **Save migrations (`src/core/SaveData.ts`):** new **top-level** `SaveBlob` fields are backfilled by the `{ ...defaults(), ...parsed }` spread; **nested** objects (`settings`/`analytics`/`upgrades`) need an explicit deep-merge line in `load()` ‚Äî omitting one caused a real production crash (recordDeath on a legacy save). Always add a "legacy save missing field" migration test.
- **No service worker** (deliberate ‚Äî the stale-cache culprit). Vite content-hashes assets; `vercel.json` sets `immutable` on `/assets/*` and `max-age=0, must-revalidate` on `index.html`.
- **Live verification via `window.__game`:** WebGL screenshots are flaky; a hidden preview tab **stalls the async audio loader (~78‚Äì88%)** ‚Üí alias missing audio keys from a loaded buffer (`g.cache.audio.add(key, g.cache.audio.get(loadedKey))`) so scenes boot; **HMR keeps stale scene instances** ‚Üí full page reload before testing new fields; force touch with `Object.defineProperty(navigator,'maxTouchPoints',{value:5,configurable:true})` before `scene.start('Game')`; Phaser TileSprites clone textures under UUID keys (probe the field, not the texture key); `body.reset(x,y)` syncs the Arcade body immediately (plain `setPosition` doesn't until a physics step).
- **Controls:** desktop = keyboard (run/variable-jump/double-jump/wall/dash/fast-fall). Mobile = two-thumb ‚Äî hold left half = floating run joystick (analog, pull down = fast fall); tap right half = jump (hold = higher, tap-in-air = double, again = triple; mobile-only triple via `Player.maxJumps`); DASH + pause buttons. One shared `Player.update()` reads a single `InputState.runAxis` (¬±1) so desktop/mobile feel identical.
- **APK build (Windows):** Android Studio's bundled `jre` is Java 11 ‚Üí Android Gradle Plugin rejects it; use JDK 21 at `C:\Program Files\Android\openjdk\jdk-21.0.8`. `cd android && JAVA_HOME=<jdk21> ANDROID_SDK_ROOT=%LOCALAPPDATA%\Android\Sdk ./gradlew assembleDebug --no-daemon` (~25s incremental). Full steps + Play Store AAB path in `MOBILE.md`; Vercel connect + cache-test checklist in `DEPLOY.md`.
- **Deliberately out of scope**: 3-star phase ratings, per-phase timer/best-time, landscape orientation, left/right d-pad buttons. (Level select shipped in v0.11.0 as an optional companion mode alongside endless/daily ‚Äî the earlier "endless-genre conflict" reasoning against it no longer applies.)
- **Workflow:** brainstorming ‚Üí writing-plans ‚Üí subagent-driven-development (fresh implementer subagent per milestone + a holistic reviewer), each milestone live-verified via `window.__game`. Specs in `docs/superpowers/specs/`, plans in `docs/superpowers/plans/` (these live in the **outer workspace repo**, not this repo).
- **Tester distribution:** Gmail blocks `.apk` attachments ‚Üí host on GitHub Releases and email the download link (draft to `cleonwheatley@gmail.com`; the Gmail connector only creates drafts, the user sends).
- **Playwright gameplay-input trap:** zero-delay `keyboard.press()` fires keydown+keyup in one tick ‚Äî the variable-height jump-cut consumes the jump before physics registers airtime, so jumps/dashes silently no-op. Use `{ delay: 40 }` on gameplay key presses.
- **Playwright touch-start trap:** the Menu's tap-to-start zone is a 600√ó365 rect ‚Äî taps below y‚âà365 of the canvas hit dead space and the run never starts. Tap at ~30% height and poll for `Game.player` to exist rather than fixed-waiting.
- **`window.__game` is DEV-ONLY** (`import.meta.env.DEV` in main.ts) ‚Äî production builds strip it; e2e specs that read it must run against the dev server (the Playwright config does).
