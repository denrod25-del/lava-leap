# Lava Leap вҖ” Session Handoff

A snapshot of project state and hard-won context, so a fresh session (or another dev) can continue without re-deriving everything.

_Last updated: 2026-07-16 (v0.12.0)._

## What it is
**Lava Leap** вҖ” an endless vertical climber (Phaser 3 + TypeScript + Vite + Vitest). Climb procedurally-generated platforms, outrun rising lava; score = height + coins. Own git repo (`master`), public at **github.com/denrod25-del/lava-leap**. Also packaged as an Android app via Capacitor and deployed to the web on Vercel.

## Current state вҖ” v0.12.0 "Power-up Pass"
- **v0.12.0 = a power-up economy pass**, three coordinated changes: much higher spawn rates with the rocket (orange boost) weighted as the most common kind, a full visual redesign of the pickups, and ~20% slower lava.
- **Spawn-rate increase:** `HAZARD.powerupChance` 0.03 вҶ’ 0.20. Initially shipped as 0.06 ("2x"), but the measured reality was ~0.1вҖ“0.3 power-ups in a typical run's first 2000px вҖ” doubling a near-zero rate was imperceptible; 0.20 lands ~1.8 power-ups per first-2000px (measured). **Lesson: for frequency tuning, measure the per-run FELT rate (events per typical run), not the per-platform probability вҖ” a multiplier on a tiny base is invisible.**
- **Weighted kind mix:** `rollPowerup` (`src/core/hazardRules.ts`) now picks from `POWERUP_WEIGHTS` вҖ” rocket 0.4, shield/magnet/slowlava 0.2 each вҖ” instead of a uniform 4-way split. Still exactly 2 RNG draws per roll (the weighting only remaps the second draw), so the per-seed platform stream is unchanged; only kind assignments shift.
- **The unlock-gate arc (built, then removed вҖ” kept here as history):** the original v0.12.0 design gated the rocket behind height 4000 in endless/daily (`rocketUnlockHeight` threaded through `LevelStream`вҶ’`LevelGenerator`, later softened to convert gated rockets into shields). Play-testing rejected it twice: the gate made the boost invisible in normal play, and the shield-conversion made blue dominate вҖ” the opposite of the "more orange" the user wanted. All gate machinery was removed before ship (no trace remains in code; `git log` commits `aa00bc2`/`1996d5f`/`6c98e2d` hold the history). Two real bugs from that arc worth remembering anyway: the naive `height < Infinity` check is always true (an Infinity-default gate must use `Number.isFinite`), and discard-style gating silently cuts total density.
- **Lava slowdown (~20%):** `src/entities/Lava.ts` `BASE_SPEED` 55 вҶ’ 44, `SPEED_PER_HEIGHT` 0.012 вҶ’ 0.010 (zone multipliers untouched). Direct play-feedback change; note high scores will trend higher vs. the pre-0.12 leaderboard era.
- **Pickup visual redesign (`PowerupController`):** each pickup is now a Container holding a circle plus a hand-drawn Graphics icon per kind, with a pulse tween вҖ” **zero new PixelLab art** (trial still has 1 generation banked). Two real bugs caught and fixed during review: (1) the infinite-repeat pulse tween was never killed on container destruction вҖ” leaked a live tween + retained object per pickup; (2) tweening the CONTAINER's scale made the Arcade body oscillate ~8вҶ’9px (`Body.updateBounds` reads container scale each preUpdate) вҖ” fixed by tweening the **children** instead, so the collision footprint is constant by construction. Teardown consolidated into a single `removePickup(id)` helper (despawn, collect, and shutdown all route through it).
- **Tests:** 214 unit + 23 e2e green; typecheck + build clean; `0.12.0` confirmed in the production bundle.

## Previous state вҖ” v0.11.0 "Levels Mode"
- **v0.11.0 = a real 4-level campaign**, layered on top of endless with almost no new machinery. `src/core/levels.ts` defines `LEVELS: LevelDef[]` (`id`, `title`, `zoneIndex`, `startHeight`, `bossTriggerHeight`, `bossTemplateId`) вҖ” 4 entries, each spanning exactly 1000px (`level-1` 0вҶ’1000, `level-2` 1000вҶ’2000, `level-3` 2000вҶ’3000, `level-4` 3000вҶ’4000) and a pure `isLevelUnlocked(levelId, clearedIds)` (sequential: level N unlocked iff level N-1 is in `clearedIds`). SaveData gained `levels: { cleared: string[] }` (deep-merge + array-guarded like every other nested field).
- **The spawn-offset trick is why this was cheap:** every height-driven system in the game (`zoneForHeight`, the generator's difficulty ramp, `StoryProgress.onHeight`, `RelicPlanner`) already computes purely from `TUNING.groundY - player.y`. `LevelGenerator`/`LevelStream` gained one optional constructor param (`startHeightOffset`, default 0 вҖ” endless/daily are byte-identical) that shifts the FIRST platform's `y` and therefore the player's spawn height. Every other system just sees a bigger `heightClimbed` on frame one and behaves correctly with zero level-awareness of its own.
- **`GameScene` reused almost entirely:** `init(data)` now accepts an optional `levelId`, resolved to a `LevelDef` (`this.levelDef`). When set: background/zone/spawn/`LevelStream` all key off `startHeightOffset` instead of hardcoded zone-0/height-0; the boss trigger in `update()` branches on `this.levelDef` вҖ” level runs check their OWN `bossTriggerHeight`/`bossTemplateId` (never calling `bossBoundaryCrossed`), so `BOSS_BOUNDARIES`/`boss.ts` are **completely untouched** by this feature (`tests/boss.test.ts` still locks the endless array to `[1000,2000,3000]`, proving it). A 4th boss template (`titan-4`, hand-authored + `validateChunk`-checked) exists ONLY as a level-mode finale вҖ” endless mode still has just 3 boss encounters.
- **`completeLevel()`** (new `GameScene` method, fired from the `bossPhase 'end'` handler when `this.levelDef` is set): bank coins via the existing `endRunBookkeeping`, push the cleared id into `save.levels.cleared` (idempotent), route to `GameOverScene` with `{result:'cleared', levelId, nextLevelId}` вҖ” through `Cutscene` first if one's pending, exactly like a normal death does. **Leaderboard-gating fix:** `triggerDeath`'s auto-submit block now reads `if (leaderboard.enabled && !this.levelDef)` вҖ” a death INSIDE a level no longer contaminates the endless leaderboard (this was a real bug the spec missed; caught while writing the plan, not by the design doc).
- **`GameOverScene`** branches on `data.result === 'cleared'`: cyan "LEVEL CLEAR" heading, "Press SPACE for the next level" (only if `nextLevelId` exists вҖ” Level 4 shows just "ESC вҖ” Level Select"), no leaderboard rank block ever (levels never set `playerId`). Dying inside a level restarts that SAME level (`levelId` now threads through the retry call) rather than falling back to endless, matching "restart from the top."
- **`LevelSelectScene`** (`'LevelSelect'` key, Menu hotkey `K`): the established Journal/Shop vertical-list idiom вҖ” 4 rows, `[CLEARED вҳ…]` / `[PLAY]` / `[LOCKED рҹ”’]` computed live from `save.levels.cleared` via `isLevelUnlocked`. Clicking/entering a locked row is a no-op (no scene change, no error).
- **Version-bump ordering note (confirmed again this release):** `e2e/levels.spec.ts` seeds `lastSeenVersion: '0.11.0'` вҖ” that seed only resolves to `Menu` (not an auto-redirect to `Changelog`) once `package.json`'s version actually matches. Bumped the version + changelog entry BEFORE running the new e2e spec, same pattern the v0.10.0 handoff already documents below.
- **Tests:** 213 unit + 23 e2e green; typecheck + build clean; `0.11.0` confirmed in the production bundle.
- **Correction to a stale claim below:** the "Deliberately out of scope" list at the bottom of this doc previously said "level select" was excluded as an endless-genre conflict. That decision was reversed this release вҖ” levels mode is now shipped as an optional companion mode alongside endless/daily, not a replacement. See that section for the corrected line.

## Key technical facts / gotchas
- **Architecture:** pure logic in `src/core/` (never imports Phaser; unit-tested); Phaser in `src/scenes/` + `src/entities/`. Tuning in `src/tuning.ts`. In `GameScene` the event hub is `this.gameEvents` (Phaser owns `this.events`); input source is `this.inputSrc`.
- **Build info:** `vite.config.ts` `define`s `__APP_VERSION__` (from `package.json` via `readFileSync` вҖ” tsconfig has no `resolveJsonModule`, `moduleResolution: bundler`), `__BUILD_ID__` (git short hash via `execSync`, try/catch вҶ’ `'dev'`), `__BUILD_DATE__`; `src/build-globals.d.ts` declares them; `src/core/buildInfo.ts` re-exports with vitest fallbacks. **Bump `package.json` version each release** вҖ” every on-screen label reads it automatically.
- **Save migrations (`src/core/SaveData.ts`):** new **top-level** `SaveBlob` fields are backfilled by the `{ ...defaults(), ...parsed }` spread; **nested** objects (`settings`/`analytics`/`upgrades`) need an explicit deep-merge line in `load()` вҖ” omitting one caused a real production crash (recordDeath on a legacy save). Always add a "legacy save missing field" migration test.
- **No service worker** (deliberate вҖ” the stale-cache culprit). Vite content-hashes assets; `vercel.json` sets `immutable` on `/assets/*` and `max-age=0, must-revalidate` on `index.html`.
- **Live verification via `window.__game`:** WebGL screenshots are flaky; a hidden preview tab **stalls the async audio loader (~78вҖ“88%)** вҶ’ alias missing audio keys from a loaded buffer (`g.cache.audio.add(key, g.cache.audio.get(loadedKey))`) so scenes boot; **HMR keeps stale scene instances** вҶ’ full page reload before testing new fields; force touch with `Object.defineProperty(navigator,'maxTouchPoints',{value:5,configurable:true})` before `scene.start('Game')`; Phaser TileSprites clone textures under UUID keys (probe the field, not the texture key); `body.reset(x,y)` syncs the Arcade body immediately (plain `setPosition` doesn't until a physics step).
- **Controls:** desktop = keyboard (run/variable-jump/double-jump/wall/dash/fast-fall). Mobile = two-thumb вҖ” hold left half = floating run joystick (analog, pull down = fast fall); tap right half = jump (hold = higher, tap-in-air = double, again = triple; mobile-only triple via `Player.maxJumps`); DASH + pause buttons. One shared `Player.update()` reads a single `InputState.runAxis` (Вұ1) so desktop/mobile feel identical.
- **APK build (Windows):** Android Studio's bundled `jre` is Java 11 вҶ’ Android Gradle Plugin rejects it; use JDK 21 at `C:\Program Files\Android\openjdk\jdk-21.0.8`. `cd android && JAVA_HOME=<jdk21> ANDROID_SDK_ROOT=%LOCALAPPDATA%\Android\Sdk ./gradlew assembleDebug --no-daemon` (~25s incremental). Full steps + Play Store AAB path in `MOBILE.md`; Vercel connect + cache-test checklist in `DEPLOY.md`.
- **Deliberately out of scope**: 3-star phase ratings, per-phase timer/best-time, landscape orientation, left/right d-pad buttons. (Level select shipped in v0.11.0 as an optional companion mode alongside endless/daily вҖ” the earlier "endless-genre conflict" reasoning against it no longer applies.)
- **Workflow:** brainstorming вҶ’ writing-plans вҶ’ subagent-driven-development (fresh implementer subagent per milestone + a holistic reviewer), each milestone live-verified via `window.__game`. Specs in `docs/superpowers/specs/`, plans in `docs/superpowers/plans/` (these live in the **outer workspace repo**, not this repo).
- **Tester distribution:** Gmail blocks `.apk` attachments вҶ’ host on GitHub Releases and email the download link (draft to `cleonwheatley@gmail.com`; the Gmail connector only creates drafts, the user sends).
- **Playwright gameplay-input trap:** zero-delay `keyboard.press()` fires keydown+keyup in one tick вҖ” the variable-height jump-cut consumes the jump before physics registers airtime, so jumps/dashes silently no-op. Use `{ delay: 40 }` on gameplay key presses.
- **Playwright touch-start trap:** the Menu's tap-to-start zone is a 600Г—365 rect вҖ” taps below yвүҲ365 of the canvas hit dead space and the run never starts. Tap at ~30% height and poll for `Game.player` to exist rather than fixed-waiting.
- **`window.__game` is DEV-ONLY** (`import.meta.env.DEV` in main.ts) вҖ” production builds strip it; e2e specs that read it must run against the dev server (the Playwright config does).
